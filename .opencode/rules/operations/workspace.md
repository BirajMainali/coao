# Workspace Management

## Purpose

Every work item creates an isolated workspace where all collaboration, progress tracking, and deliverables are managed.

---

## Layout

```
.coao/<type>s/<slug>/
├── context.md              - Shared ticket
├── decisions.md            - Decision log
├── relationships.md        - Cross-work-item relationship manifest
├── research.md             - Research findings (True Researcher output)
├── research-brief.md       - Context bridge for True Researcher (SA/PO authored)
├── product-owner/          - PO artifacts (requirements, stories)
├── solution-architect/     - SA artifacts (design, research)
├── software-engineer/      - SE artifacts (implementation notes)
├── qa-engineer/            - QA artifacts (test plans, results)
└── attachments/            - Supporting files
```

All agents contribute to `context.md` and `decisions.md`. Role-specific work lives in the agent's own directory.

---

## context.md

context.md is the **shared ticket** — the single document every agent reads to understand the work item at a glance. Every agent reads it on entry and updates it before exit.

### Sections

| Section | Owned by | Purpose |
|---------|----------|---------|
| Summary | PO | What is being built and why |
| Workflow | Shared | Agent sequence, current phase, timeline |
| Requirements | PO | What needs to be delivered |
| Design Overview | SA | Chosen architecture, key decisions |
| Implementation Summary | SE | What was built, notable decisions |
| Validation Summary | QA | Test results, sign-off status |
| Knowledge Outcome | Last agent | What knowledge was promoted or why none |
| Related Work Items | Shared | Cross-references to related work items with relationship types |
| Related Knowledge | Shared | Links to ADRs, patterns, standards referenced |
| Current Status | Shared | Who is working, progress, blockers |
| Open Questions & Risks | Shared | Unresolved items and known blockers |
| Next Actions | Shared | What should happen next |

### Boundary Rule

If every agent needs it at a glance → `context.md`. If one role owns it → their directory. Duplication is a violation.

---

## Role Directories

Agents decide what to put in their directory. No prescribed filenames or structure. The boundary rule is the only constraint.

---

## relationships.md

`relationships.md` is the cross-work-item manifest. It declares typed links to other work items, ADRs, and knowledge entries.

Agents create or update `relationships.md` during planning (required for feature, project, spike; optional for fix). Agents check `.coao/context-graph.md` before starting work to discover related items.

See `.opencode/templates/relationships.md` for the schema and relationship types.

## research-brief.md

`research-brief.md` is the context bridge for the True Researcher. Authored by SA or PO when a spike or research task is created. It provides domain terms, constraints, and the research question without exposing project code — enabling TR to research without violating the Eliminate X constraint.

See `.opencode/templates/research-brief.md` for the template.

## Context Graph Root

`.coao/context-graph.md` is the auto-generated index of all relationships across workspace manifests. Generated by:

```bash
bash .opencode/scripts/generate-context-graph.sh
```

Regenerate whenever a `relationships.md` is created or updated. The graph root is the single entry point for any agent to discover what connects to what.

---

## Workspace Lifecycle

Create → Use (continuous context.md updates) → Finalize → Archive

Before archiving, run the context graph generator to capture all relationships. Knowledge candidates go to `knowledge/candidates/` at repo root — never in a workspace.
